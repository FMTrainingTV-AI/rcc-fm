## Tried / Learned / Decided

Open-ended start: Joe wanted a 5th sibling starter for project-management work, broader than FileMaker. Pulled two Granola transcripts that informed everything — Charlie Bailey (Codence, AI in PM) for the operational rhythm, Wei Hao (Direct Impact, Magic-the-Gathering scope control) for the planning vocabulary. Charlie gives the daily heartbeat; Wei Hao gives the scope language.

Three decisions did most of the work:

1. **Sessions, not changelog.** Joe's instinct was that the existing changelog pattern wasn't capturing the *why*. Merged what + why into per-day sessions. Avoids Charlie's "single giant log" failure mode because per-day caps context size. `decisions/` stays separate, opt-in, for durable choices retrievable by topic. Briefly considered keeping them as one folder — Joe pushed for the merge; right call.

2. **Skeleton, not MVP.** Wei Hao explicitly retires MVP terminology because it's been corrupted into "phase one deliverable." Skeleton is more honest. The `skeleton.md` template is the always-here planning artifact; it scales with content (paragraph for change-requests, full doc for engagements). This single move solved Joe's "I don't want two versions" problem.

3. **Milestones as opt-in expansion, not default unit.** Same scaffold from change-request through fixed-bid. Milestones sprout when the path to the skeleton is too long. Sessions stay unified across milestones — Charlie's single-narrative principle preserved.

Also locked: **companion mode** (PM folders drop into datacraft-fm/web/sass/agent alongside the shared canonical — `prototypes/` stays standalone-only since code starters already have `dev/`/`webviewer/`/`web/`). **Capital-P naming convention** — `datacraft-Project` as placeholder; replace "Project" with project/client name on clone. Diverges from sibling lowercase by design; the casing signals the fillable slot.

Built the HTML artifact first (Joe wanted "why I did that" as a reference), then executed the scaffold from the artifact. Order mattered — seeing the design whole made the implementation mechanical. The artifact + this session entry together form the durable record of the design.

Not yet a git repo. Deferred — `_starters/` is Dropbox-tracked. `git init` happens at clone time for a real project.

## Addendum — Active-intent layer added later same day

After the initial scaffold landed, Joe shared Matt Maher's YouTube piece on agent workflows ("AI Has Started Building AI — and It's Already Here," 2026-06-08). Maher named a gap the original two sources didn't explicitly cover: **agents nail the *what* and let the *why* go.** Joe asked whether incorporating an active-why layer was worthwhile. It was.

**What was added:**

- `docs/TASKS.md` — Current items now carry a one-line `Why` + `Done-when`. Items in Next/Waiting/Backlog don't need this until promoted.
- `docs/sessions/_template.md` — new `## Intent` section at the top (today's push, why, done-for-today, not-in-scope), set at start of substantive work. New `## Intent vs. outcome` section at the end, filled by `stepping-away`.
- `.claude/skills/whats-next/SKILL.md` — output now includes a drafted Intent block ready to paste into today's session entry.
- `.claude/skills/stepping-away/SKILL.md` — end-of-day check compares Intent to actual outcome; drift is named explicitly (deliberate pivot vs. unnoticed drift, with cause).
- `CLAUDE.md` — new "Active-Intent Layer" section explaining the four temporal layers (macro why / what / active why / past why) and where each lives.
- `docs/_design/2026-06-06-starter-design.html` — Maher added as a third source; "Three core design moves" became "Four"; the Origin section now has a Maher callout box.

**Reasoning:** the skeleton holds macro why (project-level). TASKS holds the queue. Sessions hold past why. Without an active-why layer at the session level, the agent has no anchor it can read on every tool call — Maher's "the why quietly goes." The Intent block is small (30s to write), travels with the agent through the day, and the end-of-day intent-vs-outcome check is the drift-catching mechanism.

**Why this was right to add same-session, not deferred:** the starter hasn't been cloned yet. Baking the pattern in now is cheaper than retrofitting later, and it strengthens the design rationale visibly. The framing now reads coherently as three sources covering ops + scope + orientation, not two sources with an obvious missing piece.

**Housekeeping:** YouTube transcript VTT files pulled to `/tmp` during research were deleted. The design HTML cites Maher's video URL; no transcript stored in the starter (Joe's call — keeps the template clean of session-incidental content).

## Addendum 2 — Lightening pass

Same session. After the Intent layer landed, Joe asked for an honest assessment: had the starter gone too heavy? Answer was: structure is light, but documentation and skill count had drifted heavy. Specifically, `CLAUDE.md` had grown to ~280 lines reading like a methodology textbook, and the four-skill table presented `scope-trade` and `flavor-check` as ongoing rituals when they're really conversation patterns.

Joe agreed and explicitly named the trim: "scope trade and flavor check, I don't even need skills for that. I think we can have that kind of conversation."

**What was trimmed:**

- **Deleted** `.claude/skills/scope-trade/` and `.claude/skills/flavor-check/` (whole folders, including their `SKILL.md` files).
- **CLAUDE.md** rewritten from ~280 lines to 133 lines. Compact repo layout. Compact convention prose. The four-skills table is now a two-skills table. Wei Hao's scope-trade and flavor-check principles survive as one-line entries in "Working conventions" — handled in conversation, documented in the session entry, not formalized as skill files.
- **Intent block template** loosened from 4 structured bullets to 2–3 sentences of plain prose with an inline example. Same information, no form to fill.
- **HTML design artifact** updated: folder tree now shows two skills; "Four skills" section is now "Two skills + two patterns" with an explicit "dropped, and where they went instead" subsection. Companion-mode prose updated to "two skills."
- **`docs/skeleton.md`** updated to reference scope-trade and flavor-check as conversational principles, not skills.

**Reasoning:** the principles named by Wei Hao are real and worth practicing. But formalizing them as skill files implied a discipline they don't need — Joe doesn't run a monthly flavor-check ritual, he has a flavor-check *conversation* when something feels off. The skill file made the practice feel rigid. Stripping it to a one-line convention keeps the principle accessible without imposing the form.

This is also the scope-trade principle in practice — applied to the starter itself. The question was "if this Intent-layer push has added weight, what comes out?" The answer: two skills that earned their place on idea, not on use.

**Net effect:**

- File count: 50 → 48 (two skill folders removed, no new files)
- CLAUDE.md: 280 → 133 lines (~53% reduction)
- Skills: 4 → 2 (both daily, no situational)
- Friction at session start: 4-bullet form → 2–3 sentences prose

The structure didn't change. The vocabulary got lighter.

**Continuation pass — same direction.** Joe asked to extend the trim to "everything else suggested." Done in one more pass:

- **`stepping-away` SKILL.md** — 81 → 35 lines. Same five-step checklist, half the prose.
- **`whats-next` SKILL.md** — 82 → 46 lines. Plus fixed an embedded leftover (the old 4-bullet Intent template had survived inside the skill file — now matches the loosened prose form everywhere).
- **`docs/TASKS.md`** — 39 → 29 lines. Stripped the explanatory pre-section; the pattern lives inline in one code block.
- **`docs/sessions/_template.md`** — 39 → 25 lines. Each section instruction is one sentence now, not three.

Folder READMEs (each 14–34 lines) and milestone template files (23–34 lines each) were spot-checked and left alone — they're already tight for their job.

Final state: CLAUDE.md 133 lines, two skills, four-section session template, prose Intent block, every reference to deleted skills cleared. The starter as a *read thing* is now genuinely light.

## Addendum 3 — `_pm/` restructure + paired `datacraft-Project-FM/`

Same session. After the second lightening pass, Joe looked at the top-level structure (8 PM folders + docs/ + resources/ + 3 files = 11 root items) and saw the future problem: when this drops into a code starter that already has its own busy root (e.g., `datacraft-fm` with `plugin/`, `schema/`, `webviewer/`, `web/`, `dev/`, `scripts/`), the combined root becomes a 20-item mess.

Joe shared two real-world projects (`/Users/joe/Dropbox/DC_Code/_Exoteric/Trufflepig/` and `/Users/joe/Dropbox/DC_Code/RCC_SB-SOS/`) to ground the discussion. Both had `docs/` folders with 16–17 items each — TASKS + changelog + 12+ ad-hoc Claude-generated artifacts (emails, specs, plans, mockups, meeting notes) all piled into one folder. The empirical reality of what Joe's `docs/` becomes over months.

**Decisions:**

1. **Wrap PM into `_pm/`** — underscore prefix matches Joe's `_app/`/`_ws/` convention in `datacraft-sass`. Sorts to top; signals "container, not working folder."
2. **Move TASKS and sessions INTO `_pm/`** — Joe's stronger position. Initial proposal kept TASKS/sessions in `docs/` and moved only the six PM-specific subfolders. Joe pushed further: "I like the idea of the PM and tasks being the tasks, and everything being in PM." Sessions, decisions, TASKS, and the skeleton all moved into `_pm/`.
3. **`docs/` survives but becomes the catch-all** for meta and ad-hoc Claude-generated stuff that isn't PM workflow. Notes, quirks, design history. Matches Joe's empirical pattern from Trufflepig and RCC.
4. **Rename `datacraft-fm/` → `datacraft-Project-FM/`** — establishes a naming convention: `datacraft-Project-X` is a flavored combination starter (PM + that flavor's code tooling). Future: `-Web`, `-SaaS`, `-Agent`.

**What was executed:**

- `datacraft-Project/`: moved 6 folders (artifacts, bridges, deliverables, milestones, prototypes, requirements) plus 4 things out of docs/ (TASKS.md, skeleton.md, sessions/, decisions/) into `_pm/`. Wrote `_pm/README.md`. Updated CLAUDE.md, README.md, both SKILL.md files, .gitignore, all cross-references in folder READMEs.
- `datacraft-fm/` → `datacraft-Project-FM/`: renamed the folder. Created `_pm/` and moved `docs/TASKS.md` + `docs/changelog/` into it (rename to `_pm/sessions/`). Copied the rest of `_pm/` scaffold from `datacraft-Project/` (skeleton template, decisions, milestones, requirements, artifacts, prototypes, deliverables, bridges). Copied stepping-away and whats-next skills into `.claude/skills/`. Updated CLAUDE.md significantly: new layout tree, new "Project Management Layer" section replacing the old "Task Tracking" section, "New Project Setup" steps updated to reference `_pm/`. FM-specific content (DDR pipeline, plugin model, four database access modes, web viewer playbook) preserved verbatim.
- `_starters/CLAUDE.md` (parent index): rewrote to reflect 5 starters, the capital-P naming convention, `_pm/` as the canonical PM scaffold, and deprecation of the old `task-changelog/` canonical.

**The split — final state:**

```
datacraft-Project/  (pure PM)               datacraft-Project-FM/  (PM + FM)
├── docs/  (meta + catch-all)               ├── docs/  (FM reference + catch-all)
├── resources/  (canonical taxonomy)        ├── resources/
├── _pm/  (PM apparatus)                    ├── _pm/  (same PM apparatus)
└── .claude/skills/                         ├── plugin/  webviewer/  web/  dev/
                                            ├── schema/  scripts/
                                            └── .claude/skills/  (FM + PM)
```

`_pm/` is portable — drop into any starter. Same shape in both, no path changes needed inside.

**Still deferred:** retrofitting `_pm/` into the other three starters (`datacraft-web/`, `datacraft-sass/`, `datacraft-agent/`) and into Joe's two live projects (Trufflepig, RCC_SB-SOS). Each is a mechanical move once Joe decides to do them.
