Starter design — datacraft-Project

A project-management starter that stays light and grows heavy.

Date: 2026-06-06

Constraints: Light by default. One structure that scales both ways. Preserves the task+changelog rhythm Joe already likes. Drops into the existing four code starters as a companion layer without forking.

Success looks like: A fifth sibling beside datacraft-web, -sass, -fm, -agent. Equally usable for a three-hour change request OR a six-month fixed-bid engagement. No structural decision required up front about how big the project is.

What this is NOT: A heavier methodology imposed on every project. A separate template per project size. A code framework — this is the connective tissue around an engagement, not the build.

Origin

Three sources, three layers of the same answer.

This starter is a distillation of three sources Joe encountered around the design session: Charlie Bailey on AI-enhanced project management, Wei Hao on Magic-the-Gathering-style scope control, and Matt Maher's commentary on agent workflows. They cover the daily heartbeat, the planning vocabulary, and the agent-orientation problem — and none is complete without the others.

Charlie Bailey · Codence

The operational layer.

  • Stepping-away ritual at end of day
  • Single shared session log per project
  • "What's next?" answerable from context, not from the PM
  • Ticket sync (Asana/Jira) via skill, not by hand
  • Client-facing dashboard dropped into FM web viewer
  • HTML prototypes in 30–45 min for customer validation

Wei Hao · Direct Impact

The planning vocabulary.

  • Skeleton replaces "MVP" — 5-step structured artifact
  • One in, one out for trade-offs under fixed scope
  • Flavor check — recurring drift audit
  • Test, learn, shrink — thin slice philosophy
  • Backlog should produce a coherent release, not a smaller pile

Matt Maher · on agent workflows

The agent-orientation problem.

These systems were built to execute. So they nail the what, the list of features — and they quietly let the why go. Once you really see that, the frontier stops being the code at all. It just becomes the ideation, the judgment, the deciding what's even worth building. Matt Maher — "AI Has Started Building AI — and It's Already Here," 2026-06-08

Maher's framing named a gap the first two sources didn't explicitly cover: the agent's session-level orientation. Charlie has the daily rhythm. Wei Hao has the scope discipline. But neither says where the active why — the why for the current push — lives so the agent can read it on every tool call. The starter's Intent block (top of each session entry, set by whats-next, checked by stepping-away) is the answer.

Charlie gives the daily heartbeat. Wei Hao gives the language for talking about scope. Maher names what the heartbeat is for — keeping the lead-agent oriented when execution wants to drift from intent. Together they cover ops, strategy, and orientation.

Four core design moves

The decisions that shaped everything else.

1. Sessions, not changelog.

The canonical task+changelog pattern survives, but changelog/ becomes sessions/. Per-day files capture what shipped and why we did what we did in one artifact — Charlie's running narrative without his single-file sprawl. The "stepping away" skill writes this file.

Why
Joe was already rambling thinking into the changelog; the rename matches the truth of the content.
Trade
Slight divergence from the "ledger" connotation of "changelog." Worth it — propagate the rename to all five starters.

2. Skeleton, not MVP.

Wei Hao's skeleton.md is the default planning artifact — outcome sentence, one user journey, minimum capabilities, fundamental enablers, non-negotiables. For a change-request job it's a paragraph. For a six-month engagement it's a longer doc. Same artifact, scales with content.

Why
"MVP" has been corrupted into "phase one deliverable." Skeleton is more honest about the bare-bones nature.
Trade
New vocabulary to internalize. Worth it — the term does the conceptual work.

3. Light by default. Heavy grows organically.

One scaffold. Charlie's milestones/ folders become an opt-in expansion pattern, not the default unit. Small projects never sprout them. Big projects do — when the path to the skeleton is too long for one go.

Why
"Pick light or heavy at start" loses every time. Same scaffold from change request through fixed-bid avoids the fork.
Trade
Asks the user to recognize when complexity warrants sprouting milestones. Documented in CLAUDE.md with the rule of thumb.

4. Active intent — the agent's anchor for the day.

The skeleton holds the macro why. Sessions hold the past why. TASKS holds the queue. The fourth layer — added after Maher's framing exposed the gap — is the active why: a small ## Intent block at the top of each session entry that names today's push, why it matters, what done-for-today looks like, and what's explicitly not in scope.

Why
Agents nail the what and quietly let the why go. Without an anchor the agent can read on every tool call, drift accumulates silently. The Intent block is that anchor; the Intent-vs-outcome check at end-of-day is the drift-catching mechanism.
How
whats-next drafts the Intent block at session start. stepping-away compares it to what shipped at session end. TASKS.md Current items carry a one-line Why + Done-when that travels into the Intent block.
Trade
30 seconds of writing at session start. Skipped explicitly for quick fixes. The friction is paid back the first time it catches drift.

Shared canonical

What changes in task-changelog/.

The canonical source-of-truth at _starters/task-changelog/ gets renamed and expanded. All five starters then propagate.

WasBecomesReason
task-changelog/ task-sessions/ Name matches the broader scope. The folder still contains TASKS.md as the spine.
changelog/ sessions/ Per-day files now capture shipped + thinking. Same per-day pattern Joe already likes.
changelog/_template.md sessions/_template.md Adds Shipped / Tried-Learned-Decided / Open-threads sections.
— decisions/_template.md New. Opt-in ADR pattern for durable choices that need to outlive a per-day file.

Same-commit rule extends: when work ships, TASKS.md + today's sessions/ entry + (if any) decisions/ entry move together in one commit.

Two skills + two patterns

Rituals: only what earns daily use.

The starter ships two skills, both daily. Two earlier candidates (scope-trade and flavor-check) were considered and dropped — they're conversation patterns, not rituals, so they live in CLAUDE.md's working conventions as principles rather than skill files.

stepping-away daily — close

End-of-day ritual. Joe currently rambles to do this manually. The skill formalizes the checklist so he doesn't have to. Compares today's Intent to what shipped (drift catch).

Reads
TASKS, today's session entry (with its Intent block), git log since midnight, the conversation
Writes
Today's sessions/YYYY-MM-DD.md — Shipped, Tried/Learned/Decided, Intent-vs-outcome. Updated TASKS. Optional decision.

whats-next daily — open

Morning open. Future-Joe four days later or a teammate cold-starting gets a grounded answer plus a drafted Intent block ready to paste.

Reads
Last 1–3 session entries, current TASKS (with Why/Done-when), skeleton, recent decisions
Writes
The day's Intent block into today's session entry. Proposes 1–2 things to pick up.

Dropped — and where they went instead

scope-trade
Wei Hao's "one in, one out" for new requests. Lives in CLAUDE.md working conventions as a one-line principle: ask "if this comes in, what comes out?" and document the trade in the session entry. Not a skill — a conversation.
flavor-check
Wei Hao's drift audit. Lives in CLAUDE.md as a periodic prompt: monthly, restate the release goal in one sentence and compare to what's being built. Not a skill — a conversation.

Reason for the trim: the principles are real and important, but formalizing them as skill files makes them feel like discipline you have to perform. Two daily skills + two named conversational patterns is lighter than four skills, with no loss of substance.

Template — skeleton.md

The default planning artifact.

Wei Hao's 5-step structure, slightly tightened for prose. Lives at docs/skeleton.md on day one of any project. Empty for change-requests, fuller for engagements.

# Skeleton — <Project Name>

## Outcome
<One sentence. "After this release, <user> can <do X>.">

## The one user journey that must stay intact
<The critical path. End to end. This is the proof of value.>

## Minimum capabilities
<Only what, if removed, would break the journey.>
- ...
- ...

## Fundamental enablers
<Minimum data, roles, permissions, safety, compliance basics.>
- ...

## Non-negotiables
<Budget, timeline, capacity, regulatory constraints. The fixed walls.>
- ...

---

## Quick test for any candidate story
A story belongs in the skeleton only if it makes at least one of these true:
- Without it, the core journey cannot be completed.
- Without it, the release is not usable or safe for intended users.
- Without it, the release is not testable or releasable.
Companion mode

Dropping the PM layer into an existing code starter.

When a project IS code (FM, web, SaaS) but also needs the PM layer — most client engagements — the PM-specific folders drop in alongside the code-starter's existing structure. The shared canonical (TASKS, sessions, decisions) is already there from the rename.

datacraft-fm-acme/                   ← FM project for client Acme
├── docs/                            ← shared canonical (TASKS, sessions, decisions)
├── plugin/  webviewer/  web/        ← FM-specific code (unchanged)
├── schema/  scripts/  dev/          ← FM-specific tooling (unchanged)
│
├── milestones/                      ← PM layer dropped in
├── requirements/                      same as standalone
├── artifacts/                         (Zoom transcripts, customer docs)
├── deliverables/                      (client status dashboard)
├── bridges/                         
│
├── resources/                       ← canonical taxonomy
└── .claude/skills/                  ← 4 PM skills + any code skills

Same conventions, same skills, same rituals. prototypes/ does not drop in — the code starter's existing dev/, webviewer/, web/ already serve that purpose. The PM layer is a portable subset, not a parallel universe.

Naming convention

The capital-P is intentional.

datacraft-Project/ uses an uppercase P as a placeholder token. When starting a new engagement, replace Project with the client or project name: datacraft-Acme, datacraft-Kelsey, datacraft-SBSOS.

This diverges from the lowercase sibling convention (datacraft-web, -sass, -fm, -agent) by design. The other four are typed nouns. This one is a fillable slot. The casing makes the difference visible.

The thing I like about that is it keeps this whole memory and conversation connected with that folder, so we're not making a new folder. Joe — design session, 2026-06-06

Rename-in-place (or rename-on-copy) keeps Claude's accumulated memory tied to the project from the moment of cloning forward. Worth a one-liner in the parent _starters/CLAUDE.md so future-Joe doesn't try to "fix" the casing.

Implementation plan

What gets built, in what order.

Now — this session

  • Write this design artifact in progress
  • Build the datacraft-Project/ scaffold end-to-end: CLAUDE.md, README, all folders + READMEs, skeleton template, sessions template, decisions template, four skill scaffolds, resources taxonomy, bridges README

Deferred — separate session, after Joe reviews

  • Canonical rename: task-changelog/ → task-sessions/, expand template to capture what+why, add decisions template
  • Propagate to existing four starters: rename docs/changelog/ → docs/sessions/, port new template, update CLAUDE.md references
  • Update parent _starters/CLAUDE.md: add fifth starter, document the capital-P rename convention, document companion-mode drop-in

Order is deliberate. Build the new starter first so Joe can see it whole and react before any of the other four starters get touched. The canonical rename is mechanical work once the shape is approved.

Open + deferred

Things considered, parked, or future.

Test-learn-shrink as philosophy, not artifact.

Wei Hao's fourth practice — thin slices, story splitting, evidence-driven shrinking — lives as guidance in CLAUDE.md rather than its own folder. The starter enables it via prototypes/ + skeleton + the scope-trade conversation pattern, but doesn't impose its own splitting taxonomy.

Client-facing dashboard generation.

Charlie's pattern of generating a client-visible status dashboard (dropped into the client's FM web viewer) is the reason deliverables/dashboards/ exists. The starter documents the pattern but ships no generator — too context-dependent.

"My agent talks to your agent" async client involvement.

Charlie called this out as a future direction. Not in scope. bridges/ is currently for the human Claude↔ChatGPT PII shuttle, not for client-agent interop.

Skills repo separation.

Charlie's team maintains a shared GitHub repo of cross-project skills (with PRs). Joe has a similar setup with personal home-folder skills. This starter ships project-local skill scaffolds only — the cross-project versions stay in Joe's home folder. Documented in CLAUDE.md.

Milestone retrospectives.

When milestones are sprouted, each ends with a retrospective (lives in milestones/M0-name/retro.md). Pattern documented; not in the empty scaffold.
